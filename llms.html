⚙️ HOW LLMs GENERATE TEXT
💬LLM Bot — the magic loop:

Every word you see from ChatGPT = this loop ran once. One token at a time. A 500-word answer = ~650 loops in ~3 seconds. 🤯

1
🔢 TOKENISE

"Hello World!" splits into integer IDs: [9906, 3146, 0]
The model never reads letters — only numbers. Every word you've ever written = a list of integers to the LLM.

each number maps to...
2
🧮 EMBED

Token 9906 → [0.21, -0.87, 0.44, ...] — a 4096-number vector that encodes its meaning.
"King" and "Queen" vectors are mathematically similar. "Cat" and "Dog" are close. "Cat" and "Tax Return" are very far apart.

all tokens talk to each other...
3
👁️ ATTENTION

Every token looks at every other token simultaneously across 120 stacked layers.
"The trophy didn't fit because IT was too big" → attention figures out IT = trophy, not bag. That's contextual understanding!

scores every possible next word...
4
🎲 SAMPLE

Final layer scores all ~100K tokens at once:
"Paris" 42% "London" 18% "Berlin" 12% ... 99,997 others
Temperature=0: always picks "Paris". Temperature=1: sometimes "London". Temperature=2: picks randomly. 🎲

add token → go back to step 1...
5
🔁 REPEAT

The picked token joins the input and the whole loop runs again from Step 1. Token by token, word by word, until a special [END] token is generated.
This is called autoregressive generation — each word depends on all previous words.

🔄
Steps 1→5 repeat for every single token in the output. A 500-word answer = ~650 loops. GPT-4 runs all of this in ~3 seconds. That's why GPUs matter!
🔤 KEY CONCEPTS — HOVER ANY TERM
🔢
TOKEN

Text splits into tokens. GPT-4 knows ~100,000 different tokens. Your message = list of numbers the model actually reads.

🧮
EMBEDDING

Each token maps to a vector of numbers encoding its meaning. Similar concepts point in the same mathematical direction.

👁️
ATTENTION

The core of Transformers. Every token pays attention to ALL others at once — understands context, not just word order.

🌡️
TEMPERATURE

0 = deterministic & factual. 0.7 = creative & balanced. 1+ = unpredictable. Use 0 for coding tasks, 0.7 for writing.

📏
CONTEXT WINDOW

Max tokens processed at once. Fill it up and the model forgets the beginning of your chat. Bigger = better but costs more.

🌀
HALLUCINATION

LLMs predict likely-sounding text, not truth. They can generate fluent, confident, completely wrong answers.

💬 MISSION 2/10 🐍 PYTHON
TOKENISATION — TEXT BECOMES NUMBERS 🔢
How text splits into tokens. Why 'ChatGPT' = 2 tokens. tiktoken library. Token counting for cost control.
😤 THE PROBLEM

LLMs don't read letters or words. They read tokens — subword chunks mapped to numbers. Get tokenisation wrong and you'll over-spend on API calls, hit context limits unexpectedly, and not understand why the model 'cuts off' mid-answer.

🔢Token Bot — the surprising facts:

'Hello' = 1 token. 'ChatGPT' = 2 tokens (Chat + GPT). 'pneumonoultramicroscopicsilicovolcanoconiosis' = 13 tokens 😅

Why does this matter?
💰 Cost: OpenAI charges per token. 1M tokens ≈ $15.
📏 Limits: GPT-4 max = 128K tokens. Fill it up = model forgets start.
⚡ Speed: More tokens = slower response.

Rule of thumb: 1 token ≈ 0.75 English words. 1000 tokens ≈ 750 words ≈ 1.5 pages. 🎯

Use tiktoken to estimate cost before sending large documents. A 100-page PDF = ~50,000 tokens = $0.75 for GPT-4o input. Small but adds up at scale! Consider chunking large docs and processing section by section.

EMBEDDINGS — MEANING AS MATHS 🧮
Convert text to vectors. Semantic search. Find similar documents without keywords. The magic of meaning-space.
😤 THE PROBLEM

You have 10,000 customer reviews. User asks: 'Which reviews mention bad packaging?' Keyword search finds 'packaging'. But misses 'box was crushed', 'arrived broken', 'poorly wrapped'. Embeddings find semantic meaning — not just keywords. Same concept → same direction in vector space.

🧮Embedding Bot — the mind-bending fact:

'King' converted to a vector. 'Queen' converted to a vector. 'Man' and 'Woman' converted to vectors.

Then: King - Man + Woman ≈ Queen (in actual maths on actual vectors)!

The model learned the concept of royalty and gender from predicting text — then encoded it as GEOMETRY.

Semantic search: your question becomes a vector, each document becomes a vector, find the closest ones. That's why 'bad packaging' finds 'arrived broken'! 🎯

from openai import OpenAI import numpy as np client = OpenAI() def embed(text: str) -> list[float]: return client.embeddings.create( model="text-embedding-3-small", input=text ).data[0].embedding # 1536 numbers! def similarity(a: list, b: list) -> float: # Cosine similarity: 1.0 = identical meaning, 0 = unrelated a, b = np.array(a), np.array(b) return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))) # 🧪 Test semantic similarity v1 = embed("The packaging was terrible, box arrived crushed") v2 = embed("bad packaging") v3 = embed("great product, love it") print(f"crushed box vs bad packaging: {similarity(v1,v2):.2f}") # ~0.82 HIGH! print(f"crushed box vs great product: {similarity(v1,v3):.2f}") # ~0.12 LOW! # 🎯 Finds semantic match even with ZERO common words!

from openai import OpenAI from tenacity import retry, stop_after_attempt, wait_exponential client = OpenAI() # ⚡ STREAMING — shows tokens as they arrive def stream_chat(messages: list) -> str: full = "" with client.chat.completions.stream( model="gpt-4o", messages=messages ) as stream: for chunk in stream.text_stream: print(chunk, end="", flush=True) # print as it arrives! full += chunk return full # 💬 CONVERSATION HISTORY — remember past messages history = [{"role": "system", "content": "You are a helpful Python expert."}] def chat(user_msg: str) -> str: history.append({"role": "user", "content": user_msg}) reply = stream_chat(history) history.append({"role": "assistant", "content": reply}) # ⚠️ Trim history if approaching context limit! if len(history) > 20: history[1:3] = [] return reply # 🔁 RETRY — handles rate limits automatically @retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=4, max=10)) def reliable_chat(msg: str) -> str: return chat(msg) # auto-retries on rate limit!

The most important architectural decision. When to fine-tune. When to use RAG. The $50K question.
😤 THE PROBLEM

Your LLM doesn't know your company's internal policies, product docs, or latest data. You have two options: fine-tune the model (retrain it on your data) or RAG (search your docs at query time). Wrong choice = $50,000 wasted. Right choice = 90% accuracy in 2 hours.

🎯Architecture Bot — the restaurant analogy:

Fine-tuning = teaching a chef to permanently memorise your entire recipe book. Takes months, costs £50K, but every dish is instant.

RAG = giving a chef your recipe book to look things up. Takes 2 hours to set up, costs £50/month, and they can look up anything — even recipes added yesterday!

98% of companies should use RAG. Only fine-tune to change the MODEL'S PERSONALITY/STYLE, not its knowledge. 🎯

💬 MISSION 7/10 🐍 PYTHON
FUNCTION CALLING — LLM USES YOUR CODE 🔧
Tell the LLM what functions exist. It decides when to call them. Live data, APIs, databases — all accessible.
😤 THE PROBLEM

Your LLM gives great answers but from training data that's months old. Ask about today's weather, current stock price, or latest news — it guesses or halluccinates. Function calling: give the LLM a list of tools it can use. It decides when to call them. You get real-time accurate answers.

🔧Function Bot — the assistant analogy:

Before function calling: brilliant assistant locked in a room with no phone, no internet, no tools. Knows a lot but can't look anything up.

After function calling: same assistant, but now you slide notes under the door: 'you can call the weather API' and 'you can search the database'.

The LLM reads the note, decides WHEN to use each tool, calls it, reads the result, and incorporates it into the answer. That's function calling! 🎯

Company has 10,000 internal docs. LLM knows none. Fine-tuning = £50K. RAG in C#: SK embeds docs into Qdrant. SearchDocs [KernelFunction] retrieves at query time. Planner calls it automatically.

Always fresh. Always cited. Always accurate on YOUR data. 🎯

using Microsoft.SemanticKernel.Memory; using Microsoft.SemanticKernel.Connectors.Qdrant; // docker run -p 6333:6333 qdrant/qdrant var memory = new SemanticTextMemory( new QdrantMemoryStore("http://localhost:6333", vectorSize: 1536), kernel.GetRequiredService<ITextEmbeddingGenerationService>()); // 📚 Index docs (run when docs change) string[] docs = ["Refund policy: 30 days.", "Free shipping over £50."]; for (int i = 0; i < docs.Length; i++) await memory.SaveInformationAsync("company", id: $"doc-{i}", text: docs[i]); // 🔧 RAG plugin — Planner calls this when needed! public class RagPlugin(ISemanticTextMemory mem) { [KernelFunction, Description("Search company docs for policies and info.")] public async Task<string> SearchDocs(string query) { var sb = new StringBuilder(); await foreach (var r in mem.SearchAsync("company", query, limit: 3)) sb.AppendLine(r.Metadata.Text); return sb.ToString(); } } kernel.Plugins.AddFromObject(new RagPlugin(memory), "rag");
